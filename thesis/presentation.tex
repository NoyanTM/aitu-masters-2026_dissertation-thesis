\section{Introduction}
\begin{frame}
\frametitle{Introduction}

\begin{block}{Problem statement}
\justifying
\begin{itemize}
    \item Modern cyber conflicts as an intensified race between multiple actors (e.g., government agencies, corporations, and private groups).
    \item Internal and external components of any complex software depend on projects and products within a global supply chain.
    \item Ensuring security is only a continuous process of managing risks in the system's lifecycle rather than an absolute guarantee.
\end{itemize}
\end{block}

\begin{block}{Goal and objectives}
\justifying
Construct a secure methodology for the critical IT infrastructure, emphasizing the process of collecting and accounting for its internal resources or components for reliability and early insider threat prevention.
\end{block}

\begin{block}{Relevance and novely}
\justifying
It demonstrates the dynamics of compromised package propagation within a simulated environment and related prevention mechanisms for controlling supply chain attacks via restrictive quality gates.
\end{block}

\end{frame}

\begin{frame}
\frametitle{Literature review}
\framesubtitle{C. Compromise and trust}
\justifying
\begin{small}
Figure \ref{fig:intermediary_system_between_design_and_implementation} reveals that any operation-critical or intermediary system can affect the dependent infrastructure within trust relationship models:
\begin{itemize}
    \item Static trust - reliance on a third party without any doubt due to assumptions and distractions, considering them trustworthy.
    \item Dynamic trust - dynamic probability of being trusted by someone else and vice versa, where any mistake from both parties leads to its loss by some amount \cite{singer2020trust}.
    \item Zero trust - continuous verification evaluation of all actions and actors instead of granting full access within a protected perimeter, assuming operating in a hostile environment \cite{rose2020zero}.
\end{itemize}
\end{small}

\begin{frame}
\frametitle{Methodology}
\framesubtitle{B. Applying methodology to PyPI platform}
{\scriptsize
Figure 6 illustrates R(P, t) as a recommendation from the preventive quality gate for packages in PyPI repository.
}

\begin{flushleft}
{\scriptsize
$R(P, t) = 
\begin{cases} 
    \text{block} & \text{if } T(P) < T_{\text{min}} \\
    \text{delay}(w(T)) & \text{if } T_{\text{min}} \le T(P) < T_{\text{trust}} \text{ and } t - t_{\text{sub}} < w(T(P)) \\
    \text{approve} & \text{otherwise}
\end{cases}$ \\[0.4em]
$T(P) = f(A(P), M(P)) \in [lowest\_score, highest\_score]$
}
\end{flushleft}

{\tiny
\noindent
Where:
\begin{itemize}
    \setlength{\itemsep}{1pt}
    \setlength{\parsep}{0pt}
    \setlength{\topsep}{2pt}
    \setlength{\partopsep}{0pt}
    \item P: New incoming package instance or its version.
    \item T(P): Trust score/factor (e.g., from 0 to 1000 points) mapping analysis A(P) and package metadata M(P). It also includes temporal reasoning by systematically analyzing the history of a package across multiple versions or snapshots.
    \item $w(T(P))$: Dynamic quarantine window designed to slow down potential malware, virus, or worm propagation via time limits and restrictions.
    \item $t_{\text{sub}}$: Ingestion timestamp; $T_{\text{min}}, T_{\text{trust}}$: Threshold levels for automatic policy enforcement.
    \item Statuses: block (prevent publication for public and mark as untrusted), delay (pending additional review on hold or retries), accept (passed attestations and initial time limits).
\end{itemize}
}

\begin{columns}[T]
    \begin{column}{0.45\textwidth}
        \justifying
        \begin{small}
        Modeling discrete event simulation via SimPy library with actors and their behaviors/interactions at a regular rate (one simulation step is one day with actors performing one action per step):
        \begin{itemize}
            \justifying
            \item Developer or regular user: install packages, manage installations, report suspicious packages, and perform self-checks to decompromise.
            \item Package publisher: create new packages and release updated versions under governing constraints.
            \item Threat actor: release malicious versions by impersonating or hijacking legitimate publishers or other ways of hacking software supply chain.
        \end{itemize}
        \end{small}

\section{Conclusion}
\begin{frame}
\frametitle{Conclusion}
\begin{small}
\justifying
Limitations - our simulation is an abstract model designed to represent a limited subset of reality based on certain assumptions, which is essential for predictive analysis in scenarios that are too costly or dangerous to test on production systems.\\
\vspace{0.5cm}
Findings and contributions: 
\begin{itemize}
    \justifying
    \item Developed foundations of a system lifecycle and preventive security principles, focusing on internal threats in software ecosystems, as represented in Figure \ref{fig:supply_chain_security_during_all_lifecycle}.
    \item The importance of cybersecurity will only continue to grow as an essential component of any system development lifecycle and IT infrastructure.
    \item Iterative refinement and secure-by-design approach are ways toward securing systems against systemic vulnerabilities.
\end{itemize}
